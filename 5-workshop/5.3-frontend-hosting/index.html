Deploy React Frontend with S3, CloudFront and AWS WAF

Frontend hosting overview

In this section, the React frontend of the AI AWS Architecture Reviewer project is deployed to AWS using Amazon S3, Amazon CloudFront, and protected by AWS WAF.

The React application is built locally using Vite. After the production build is generated, the files inside the dist folder are uploaded to an Amazon S3 bucket. Amazon CloudFront is then used to distribute the website to users with better performance, better caching, and access through the CloudFront domain.

In addition, AWS WAF is associated with the CloudFront distribution to add a protection layer for the frontend. AWS WAF helps monitor, detect, and reduce potentially malicious requests before they reach the web application.

The frontend deployment flow is:

React Source Code
→ npm run build
→ dist folder
→ Amazon S3 Frontend Bucket
→ Amazon CloudFront
→ AWS WAF Protection
→ User

The frontend access flow after deployment is:

User
→ CloudFront Distribution protected by AWS WAF
→ S3 Frontend Bucket
→ React Application

The frontend bucket used in this project is:

ai-aws-reviewer-frontend-tiersteam

The CloudFront domain used to access the deployed website is:

https://d9353ayez9zar.cloudfront.net

Step 1: Build the React application

Open PowerShell and move to the project directory:

cd D:\Learning\AWS\ai-aws-architecture-reviewer

Run the build command:

npm run build

After the build process is completed, Vite generates the production files inside the dist folder.

Step 2: Create the S3 frontend bucket

Create an Amazon S3 bucket to store the production build files of the React application.

The bucket name used in this project is:

ai-aws-reviewer-frontend-tiersteam

This bucket stores files such as:

index.html
JavaScript assets
CSS assets
Image assets
Other static frontend files

Step 3: Upload frontend build files to S3

After the React application is built, upload the contents of the dist folder to the S3 frontend bucket.

Run the following command:

aws s3 sync dist s3://ai-aws-reviewer-frontend-tiersteam --delete

The --delete option removes old files from the S3 bucket if they no longer exist in the latest build. This helps prevent CloudFront or the browser from loading outdated frontend assets.

Step 4: Create the CloudFront distribution

Amazon CloudFront is used to distribute the React frontend to users.

The CloudFront distribution connects to the S3 frontend bucket as the origin. Users access the frontend through the CloudFront domain instead of accessing the S3 bucket directly.

CloudFront distribution domain:

https://d9353ayez9zar.cloudfront.net

CloudFront distribution ID:

E2JHD76RIC6AML

Step 5: Configure Origin Access Control

The S3 frontend bucket is kept private. To allow CloudFront to read files from the private S3 bucket, Origin Access Control is configured.

This configuration improves security because users cannot access the S3 bucket directly. They must access the website through CloudFront.

Step 6: Configure the S3 bucket policy

After configuring Origin Access Control, update the S3 bucket policy to allow the CloudFront distribution to read objects from the bucket.

The bucket policy allows CloudFront to perform the s3:GetObject action on the frontend files.

Step 7: Configure the default root object

In CloudFront, configure the default root object as:

index.html

This allows users to access the website by opening the root CloudFront domain without typing index.html.

Example:

https://d9353ayez9zar.cloudfront.net

Step 8: Configure React Router fallback

Because the frontend is a React Single Page Application, directly accessing routes such as /reviews or /settings may return an error if CloudFront cannot find a physical file that matches the path.

To solve this issue, custom error responses are configured in CloudFront:

403 → /index.html → 200
404 → /index.html → 200

This allows React Router to correctly handle frontend routes.

Step 9: Configure AWS WAF for CloudFront

To improve frontend security, AWS WAF is configured and associated with the CloudFront distribution.

AWS WAF works as a protection layer in front of CloudFront, helping inspect requests sent to the website. Requests with malicious patterns can be monitored or blocked based on the configured rule groups.

In this project, AWS WAF is used to protect the frontend from potentially malicious requests using AWS managed rule groups.

The managed rule groups used are:

AWSManagedRulesAmazonIpReputationList
AWSManagedRulesCommonRuleSet
AWSManagedRulesKnownBadInputsRuleSet

The meaning of the rule groups:

Amazon IP Reputation List
→ Helps detect requests from IP addresses with low reputation or a history of malicious activity.

Common Rule Set
→ Helps protect the application from common attack patterns such as abnormal requests, path traversal, or common dangerous patterns.

Known Bad Inputs Rule Set
→ Helps detect known malicious input patterns.

During the testing phase, rules can be configured in Count mode to monitor the number of requests that match the rules without blocking them immediately. After confirming that the rules do not affect valid requests, they can be switched to block mode to strengthen protection.

Frontend protection flow after AWS WAF is associated:

User
→ AWS WAF inspects the request
→ Amazon CloudFront
→ Amazon S3 Frontend Bucket
→ React Application

Step 10: Create CloudFront invalidation

After uploading a new frontend build to S3, CloudFront may still distribute cached files. To make CloudFront load the latest frontend version, create an invalidation.

Run the following command:

aws cloudfront create-invalidation --distribution-id E2JHD76RIC6AML --paths "/*"

Wait until the invalidation status changes to Completed.

Step 11: Test the deployed website

Open the CloudFront domain in a browser:

https://d9353ayez9zar.cloudfront.net

The React frontend is loaded successfully through CloudFront. At the same time, frontend access requests pass through the AWS WAF protection layer associated with the CloudFront distribution.
